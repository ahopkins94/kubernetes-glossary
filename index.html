Kubernetes Glossary


Key Terms


Container: Similar to a VM but more lightweight. They bundle and run your application. In production a container can be run on any computer that has a conterization platform
Pod: Describes an application running on Kubernetes. It can contain one or more coupled containers that make up the app e.g. an api service and a database proxy
Deployment: Defines the state of the application and Kubernetes can use this to make sure the clusters match your desired state. It can handle replica sets and roll backs
Service: Bridges mortal pods and the end user to ensure pods are never accessed directly
Secret: Provides a way to distribute secret data to pods via the deployment yaml
Ingress: Manages inbound connection to the cluster and disributes traffic to your services

Further Terms


Health Checks: If your application malfunctions, the pod and container could still be running unaware of the malfunction. Running periodic health checks can ensure traffic won't be routed to pods that do not pass the check
Persistent Volumes: Allows you to store data outside the container. Useful for when you want data to persist after the container stops e.g. running a database using persistent volumes

Docker


Docker: Container software to run containers locally or in production
Docker engine: Software to make and run docker images
Docker registry: Stores and fetches docker images. Makes an image available to Kubernetes
Dockerfile: Describes how to build a container, each container needs a dockerfile
Image: This is what is built from the dockerfile and what runs your container

Misc


Minikube: Software that spins up a kubernetes cluster on your machine
Kubectl: Software to interact with Kubernetes from the cli (equivalent of the web UI)
